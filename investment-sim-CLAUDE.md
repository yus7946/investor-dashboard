## 2. 検証コマンド(必須実行)
```bash
[ lint ] / [ typecheck ] / [ test ] / [ build ]
```
通らないものを「完了」と報告しない。

---

## 3. 進め方
- **計画→実装→検証**: 変更前に方針を明示し、一度決めたら覆さない。曖昧な要求は最も合理的な解釈を明示して進める
- **推測禁止**: バグ調査は実データ・実エラーを見てから直す
- **検証してから完了報告**: lint/test/build結果を確認してから報告。予測ロジックはとくに慎重に確認する
- **品質基準**: Apple/Microsoft/Google級。エッジケース(データ欠損、API障害、異常値)を前提に設計する
- **GAN的ディベート**: 予測ロジック・スコアリング・API連携など重要箇所は「生成→敵対的批判→改善」を最大2〜3ラウンド実施。小修正には適用しない(トークン節約と矛盾させない)

## 4. コード品質
- 可読性 > 短さ / 例外処理を握りつぶさない / `any`濫用禁止 / APIキーは環境変数のみ
- 重複は3箇所以上で共通化 / コメントは「なぜ」を書く

## 5. 禁止事項
- テストを通すためのテスト改ざん
- エラーの握りつぶし
- 確認なしの予測ロジック変更・依存追加

## 6. コミット
- 1コミット1目的。予測ロジック変更は事前明示必須

## 7. Claude Code運用
- effort: 通常`high`、予測ロジック設計/難所は`xhigh`、定型作業は`medium`
- 複雑な設計はplan mode→実行、`opusplan`があれば計画Opus/実装Sonnet

- 報告は簡潔に(変更内容・理由・検証結果のみ)

- 決めた方針は蒸し返さない

## 8. コスト制約(完全無料)
- 有料プラン・従量課金サービスの導入禁止。無料枠の上限(API呼び出し回数/レート制限)を明記し超えない設計にする
- クレカ登録必須のサービスは原則回避。使う場合は事前警告し承認を得る
- 優先順位: OSS/セルフホスト無料 → 寛大な無料枠SaaS → 有料(原則不可)
- 有料枠に触れそうと気づいたら黙って進めず即報告

## 9. プロダクトのAI人格・コンプライアンス(重要)

このアプリのAI出力は**プロの投資家エージェント**として振る舞うが、以下を必ず守る。

- **断定を避ける**: 「上がります」「買うべきです」のような確定的な投資助言・断言表現は禁止。「〇〇の可能性がある」「過去データではこう動く傾向がある」という確率・傾向ベースの表現にする
- **免責の明示**: シミュレーション結果・予測を表示する箇所には、投資判断は自己責任であり本アプリは助言ではない旨の免責表示を必ず添える
- **根拠の透明性**: 予測・スコアの根拠(使用データ、期間、前提条件)をUI上で確認できるようにする。ブラックボックスの断定的結果だけを見せない
- **リスクの併記**: ポジティブな予測を出す際は、想定されるリスク・不確実性も同じ画面内で提示する
- **口調**: プロらしい簡潔さと信頼感は保ちつつ、断定的すぎる/煽るような表現(「今すぐ買い」等)は避ける
- 予測ロジックやAIプロンプト文言の変更は3節のGANディベート対象に含め、「この表現は金融助言と誤解されないか」を自己批判する

## 10. 要確認事項
着手前に必ず確認:
- 破壊的変更・有料化しうる操作
- 予測ロジック/スコアリングの定義変更
- AI出力の口調・断定度を変える変更(9節に関わるため)
- 解釈が割れる要求

それ以外は妥当な仮定を明示し自走してよい。
